## 1. Цель и инварианты

Пересоздать бэкенд как набор изолированных модулей, общающихся через
одну общую «тупую» шину. Вся логика — внутри модулей; шина только
доставляет сообщения. Веб-интерфейс работает с теми же модулями через
тот же механизм (gateway транслирует HTTP/SSE ↔ шина).

**Инварианты (ред. 2026-06-11 — контракт больше не копируется буквально):**

| № | Инвариант | Проверка |
|---|---|---|
| И-1 | **Дизайн и UX неприкосновенны**: визуал «до пикселя», поведение интерфейса, мгновенный отклик. JS-слой данных фронта МОЖЕТ адаптироваться под улучшенный контракт — каждая такая правка согласуется и не трогает визуал | сравнение интерфейса бок-о-бок на стенде |
| И-2 | **Полнота функций**: каждая точка контракта из инвентаря (этап 0) осознанно обработана — «перенесена / улучшена / удалена», ничего не теряется молча | журнал ревизии контракта `contracts/revision.md` |
| И-3 | **Push-модель**: одно SSE-соединение, фронт без поллингов и follow-up fetch'ей (принципы 1–2); состав и формат событий — предмет ревизии | код-ревью фронт-правок |
| И-4 | **Схема `app.db` — отправная точка**: текущая схема = Alembic-миграция №1, улучшения схемы — только миграциями поверх; финальное переключение = копия прод-БД + прогон миграций | Alembic-история; прогон переключения на стенде |
| И-5 | Каждый сценарий конвейера воспроизводится на стенде | параллельный прогон + эмуляция (этап 6) |

**Философия переноса (решение пользователя 2026-06-11):** это
одновременно рефакторинг, оптимизация, улучшение и переход на шину.
Текущие точки соприкосновения фронта и бэкенда содержат костыли
(особенно статусная модель) — буквальное повторение контракта
запрещено так же, как и молчаливая потеря функциональности. Решения
принимаются **по месту**, по мере того как разбор каждого аспекта
углубляет понимание системы; ТЗ — карта, а не территория.

**Ограничитель ревизии (чтобы улучшения не расползлись):** улучшение
точки контракта должно служить одному из семи принципов проекта
(дельты вместо полных объектов → принцип 2; одна модель статусов →
принцип 7; и т.д.). Улучшение «просто красивее» — не основание,
такая точка переносится как есть.

**Баги (решение пользователя 2026-06-11):** текущее приложение не
идеально, в нём есть известные и неизвестные баги. Найденные при
переносе баги **исправляются, а не воспроизводятся** — это частный
случай ревизии: точка получает решение «улучшена», фикс согласуется
по «дисциплине находок» и заносится в `contracts/revision.md`.

**Разрешено менять:** фреймворки, внутреннюю структуру кода, модель
конкуренции (потоки → asyncio), формат логов, способ общения компонентов.

Известные причуды контракта — **первые кандидаты на ревизию** (найдены
аудитом; каждая получает решение по месту: упростить с адаптацией
фронта или перенести как есть):

- поле `state` сериала может быть строкой `"a,b"` или массивом — фронт
  обрабатывает оба (`app.js:66`);
- `chapters` в media-items отдаётся **JSON-строкой внутри JSON**, фронт
  парсит её сам (`ChapterManager.js`);
- качества Astar кодируются строкой с разделителем `;`;
- даты — ISO 8601, `Content-Type: application/json` везде.

## 2. Архитектурное решение

### 2.1 Модель развёртывания: модульный монолит, один процесс asyncio

Шина — **внутрипроцессная**, не внешний брокер. Обоснование:

| Фактор | Следствие |
|---|---|
| SQLite — однописательная БД, а схема фиксирована (И-4) | многопроцессные модули устроят борьбу за writer-lock; один процесс снимает проблему |
| Один пользователь, один сервер | Redis/NATS/RabbitMQ не дают выгоды, но добавляют операционную сложность и точки отказа |
| Интерфейс шины — ~150 строк, абстрагирован | переезд на внешний брокер позже = замена одного модуля `core/bus`, остальные не правятся |

Блокирующие операции (ffmpeg, yt-dlp, Playwright, requests к трекерам)
выполняются в executor'ах — модуль сам решает, потоки или подпроцессы,
наружу это не видно.

### 2.2 Протокол шины: конверт

```text
Envelope {
  topic:          str        # "downloads.task.finished" — иерархия через точку
  kind:           "event" | "command" | "query" | "reply"
  correlation_id: uuid       # связывает query → reply
  reply_to:       str | null # топик для ответа (только у query)
  ts:             iso8601
  payload:        JSON       # только сериализуемые данные, никаких объектов
}
```

Шина умеет ровно два действия:

- `publish(envelope)` — положить конверт;
- `subscribe(pattern) -> queue` — получить очередь конвертов по шаблону
  топика (wildcard `*` на сегмент, `#` на хвост).

**Чего шина НЕ делает (и не должна уметь никогда):** маршрутизация по
содержимому, трансформация payload, retry, persistence, приоритеты,
знание о модулях. Request/reply реализован в обвязке `core/module.py`
поверх pub/sub (reply-топик + correlation_id), шина об этом не знает.

Семантика видов сообщений:

| Вид | Грамматика топика | Подписчиков | Пример |
|---|---|---|---|
| event | факт, прошедшее время | 0..N | `slicing.task.finished` |
| command | повеление, один логический адресат | 1 | `scan.series.request` |
| query | вопрос, ожидается reply | 1 | `rules.apply` |
| reply | ответ на query | 1 (по correlation_id) | — |

### 2.3 «Шина — сигнал, БД — истина»

Ключевое правило, наследующее принцип 3 (crash-tolerance):

1. Команда конвейера сначала **персистится как задача в БД** модулем-
   владельцем (как сейчас: `download_tasks`, `slicing_tasks`, …), и
   только потом публикуется событие-будильник.
2. Потерянное сообщение ≠ потерянная задача: при старте и по таймеру
   модуль добирает свои pending-задачи из БД (текущая модель опроса
   сохраняется как страховочный механизм с большим интервалом).
3. События — для скорости реакции и для UI; состояние — только в БД.

### 2.4 Как семь принципов проекта отображаются на архитектуру

| Принцип | Реализация в новой архитектуре |
|---|---|
| 1. Push-only от бэкенда | gateway пробрасывает события шины в SSE; поллингов с фронта не появляется (И-3) |
| 2. Бэкенд — инициатор UI | ревизия SSE целится в **дельты** вместо полных объектов (давняя цель принципа — теперь в объёме работ); события самодостаточны, follow-up fetch'и не нужны |
| 3. Crash-tolerance | правило 2.3; задачи в БД, события — будильники; reconcile при старте модуля |
| 4. Дизайн неприкосновенен | И-1: визуал и UX «до пикселя»; правки JS-слоя данных — только согласованные, без изменения DOM-структуры/классов/анимаций |
| 5. Мгновенный отклик | без изменений (логика фронта сохраняется); gateway отвечает на команды сразу после персиста задачи, не дожидаясь выполнения |
| 6. Категоризированное логирование | группа логирования = имя модуля, автоматически через `core/logging`; уровни — через модуль settings |
| 7. Единый источник правды | владение данными: каждая таблица/колонка закреплена за одним модулем (раздел 6); чужие данные — только query через шину |

| `downloads` | загрузка VK-видео (yt-dlp), пул параллельных загрузок, прогресс | downloader.py, agents/downloader_agent.py |

| `trackerauth` | сессии Kinozal/RuTracker; **fetch-прокси**: выполняет авторизованные HTTP-запросы по query, сессии не покидают модуль | auth.py (кроме qBit-части — она уходит в torrents) |

Замечания к разборке текущего кода:

- **MonitoringAgent распиливается на три части:** расписание сканов →
  `scan`, relocation/верификация ФС → `library`, heartbeat — в `core`.
- **AuthManager распиливается на две:** трекерные сессии → `trackerauth`
  (как fetch-прокси, потому что `requests.Session` не сериализуется в
  конверт), qBit-сессия → приватное состояние `torrents`.
- **In-memory очередь `Agent.processing_torrents` не переносится** —
  состояние торрент-задач живёт в `agent_tasks` (это заодно чинит
  существующее нарушение принципа 3: сейчас очередь теряется при
  рестарте).

> объектов, судьба heartbeat'ов); фронт-подписчики адаптируются

## 5. Gateway: контракт HTTP/SSE

Правила:

1. Полный реестр 73 эндпоинтов снимается на этапе 0 (`contracts/`) как
   **инвентарь для ревизии**: каждая точка получает решение
   «перенесена / улучшена / удалена» с записью в `contracts/revision.md`
   (И-2). Улучшение точки влечёт синхронную согласованную правку
   JS-слоя фронта (И-1).
2. GET-эндпоинты → query в шину, ответ reply отдаётся как JSON.
3. Мутации → command (персист у модуля-владельца) + немедленный ответ
   в старом формате.
4. `/api/stream` — одно SSE-соединение; gateway подписан на таблицу
   «топик → SSE-имя» (раздел 4.1) и пробрасывает payload как есть.
5. Статика `static/`, `templates/index.html` — копия текущих, раздаются
   gateway'ем по тем же путям.

Соответствие групп эндпоинтов модулям:

| Группа (кол-во эндпоинтов) | Модуль-исполнитель |
|---|---|
| Серии: CRUD, scan, state, heartbeat, auto_scan (8) | catalog, scan, status |
| Композиция и медиа-элементы (9) | catalog (+downloads/slicing данные через колонки) |
| Нарезка: главы, фильтры, slice (6) | slicing |
| Парсер-профили и правила (12) | rules |
| Трекеры (3) | sources |
| TMDB (2) | metadata |
| История/логи (2) | catalog, settings |
| Auth + настройки + флаги (10) | trackerauth, torrents, settings |
| Очереди агентов (3) | torrents, downloads |
| Сканер: settings, scan_all (3) | scan |
| БД-viewer (3) | settings |
| Файловый браузер (1) | library |
| Прочее/служебные (≈11) | по принадлежности |

Рекомендуемый фреймворк gateway — **FastAPI/Starlette** (нативный
asyncio, штатный SSE, OpenAPI бесплатно). Это рекомендация, не
инвариант: контракт важнее фреймворка.

## 7. Структура новой кодовой базы

Новая кодовая база живёт **в том же репозитории, в ветке
`refactoring/bus`** (решение пользователя 2026-06-11; прежняя
рекомендация отдельного репозитория отменена). Дерево ветки целиком
заменяется на новую структуру; `static/` и `templates/` переносятся
из `main` как отправная точка — далее JS-слой данных правится
синхронно с ревизией контракта (визуал неприкосновенен, И-1).

```text
series-tracker/  (ветка refactoring/bus)
├── run.py                    # сборка: шина → модули → gateway
├── alembic/                  # миграция №1 = текущая схема app.db
├── core/
│   ├── bus.py                # publish/subscribe, wildcards — «тупая» часть
│   ├── envelope.py           # конверт + валидация
│   ├── module.py             # BaseModule: подписки, request/reply, heartbeat,
│   │                         #   reconcile-хук (добор pending-задач при старте)
│   ├── runner.py             # запуск/останов модулей, graceful shutdown
│   └── logging.py            # группа = имя модуля, ротируемый JSON (как сейчас)
├── modules/
│   ├── gateway/              # http/ (роуты 1:1), sse.py, static_files
│   ├── catalog/              # module.py + repository.py (свои таблицы) у каждого
│   ├── status/
│   ├── scan/
│   ├── sources/
│   │   ├── torrent/          # kinozal.py, rutracker.py, anilibria.py, …
│   │   └── vk/
│   ├── rules/
│   ├── torrents/
│   ├── downloads/
│   ├── slicing/
│   ├── renaming/
│   ├── library/
│   ├── trackerauth/
│   ├── metadata/
│   └── settings/
├── contracts/
│   ├── api_contract.md       # реестр 73 эндпоинтов (из этапа 0)
```

## 8. Этапы и критерии приёмки

| Этап | Содержание | Критерий приёмки |
|---|---|---|
| **0. Инвентаризация контракта** | снять с живого инстанса эталоны всех 73 эндпоинтов и запись SSE-потока за полный цикл конвейера — как **справочник «как было»** и реестр точек для ревизии (не как pass/fail-оракул); HTML-дампы трекеров (заготовки в `parser_dumps/`, `html_debag/`); копия `app.db` (фикстура) | `contracts/` полон: каждая точка учтена и ждёт решения; харнес сравнения готов (для точек со статусом «перенесена») |
| **1. Каркас** | core (шина, конверт, BaseModule, раннер, логирование), gateway-скелет, settings | сквозной тест: HTTP → query → модуль-эхо → reply → HTTP; логи по группам |
| **2. Инфраструктура** | trackerauth (fetch-прокси), torrents (клиент qBit), metadata, library (браузер ФС) | сессии живут/обновляются; qBit-операции против тестового инстанса |
| **3. Мозги** | rules (семантика 1:1!), sources (5 парсеров + VK), scan + SmartCollector | профили из прод-`app.db` дают идентичные результаты на эталонных названиях; парсеры — на HTML-дампах |
| **4. Конвейер** | torrents (жизненный цикл), downloads, slicing, renaming, status | каждый агентский сценарий проходит на тестовых данных; рестарт процесса в середине любой фазы → продолжение (crash-тест) |
| **5. Gateway + ревизия контракта** | каждая точка инвентаря: решение «перенесена / улучшена / удалена», реализация в gateway, синхронная адаптация JS-слоя фронта для улучшенных точек | `contracts/revision.md`: 100 % точек закрыты решениями; «перенесённые» проходят golden-сравнение; «улучшенные» проверены вручную на стенде с визуальной сверкой |
| **6. Интеграция** | стенд-контейнер: **рабочая БД с нуля** (см. «Две роли прод-БД»), **отдельный** qBittorrent (Docker внутри стенда), живые трекеры; dry-run сканов с сравнением планов против прода; **искусственная эмуляция сценариев** | глубина прогона решается по месту; чек-лист ручных сценариев пользователя |

**Две роли прод-БД на стенде (решение 2026-06-11).** Копия `app.db`
используется **только как read-only фикстура** (`tests/fixtures/`):
golden-сравнение ответов (этап 5) и верификация RuleEngine на реальных
профилях (этап 3); конвейер над ней не запускается, таблица `auth`
вычищается при снятии копии (учётки пользователь передаёт отдельно).
**Рабочая БД стенда создаётся с нуля** (Alembic-миграция №1 = текущая
схема) и наполняется вручную: учётки, 3–5 реальных сериалов, профили
правил импортируются из фикстуры. Запускать конвейер над копией
прод-БД нельзя: pending-задачи, торренты и пути медиатеки прода на
стенде не существуют — reconcile-механизм устроит шквал ошибок.

**Пересмотр схемы БД** в рамках пересоздания не выполняется (И-4):
одна и та же схема делает golden-сравнение прямым, а финальное
переключение — простым копированием свежей прод-БД. Пересмотр схемы —
пост-стабилизационная фаза через Alembic (первый кандидат —
`Series.state` ↔ `SeriesStatus`, раздел 11.3).

Эмуляция сценариев на этапе 6 (идея пользователя): держать каталог
`.torrent`-файлов одного релиза в разных стадиях жизни (например,
раздача с 2 сериями → та же с 3 → с 4) и скармливать их стенду,
воспроизводя ситуацию «вышла новая серия» детерминированно, без
ожидания реального обновления на трекере. Аналогично эмулируются
смена качества и перезаливы.

Переключение на новую систему — только после приёмки этапа 6; старый
сервер остаётся горячим резервом до решения пользователя.

## 11. Наработки из отменённого ТЗ-1, перенесённые сюда

1. **Репозитории по доменам** — реализуются естественно: `repository.py`
   внутри каждого модуля, только свои таблицы (раздел 6).
2. **«Session снаружи»** — норма транзакций модулей: одна логическая
   операция = одна транзакция, session создаёт вызывающий код модуля.
3. **Статусная модель перепроектируется в ходе пересоздания** (ред.
   2026-06-11: пользователь явно назвал статусы самой костыльной зоной
   контракта). Двойственность `Series.state` ↔ `SeriesStatus`
   устраняется в модуле status, формат статусов в API/SSE упрощается,
   фронт-чтение статусов адаптируется. **Обязательное условие
   сохраняется:** двойственность была вынужденной мерой — перед
   устранением трассировать все зависимости (бэкенд И фронт) и понять,
   зачем дубль был сделан; проект решения статусной модели
   согласуется с пользователем отдельно, до реализации модуля status.
4. **Alembic** — с первого дня: текущая схема = миграция №1; будущие
   изменения схемы (после стабилизации) — только через Alembic.
5. **Чистка по ходу** — правило «решения по находкам на месте»
   действует: дубли методов db.py при переносе в репозитории не
   копируются, каждый случай фиксируется в журнале переноса.
