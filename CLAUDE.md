## Суть проекта

**Series Tracker** — self-hosted сервис для отслеживания обновлений
сериалов на торрент-трекерах и VK-каналах с автоматической загрузкой
и приведением имён файлов к строгому формату для медиатеки. Под капотом —
визуальный конструктор правил обработки имён, конвейер агентов
(скан → загрузка → нарезка → переименование) и медиатека на файловой
системе как итоговый артефакт.

Что важно: **продукт — это не код, а медиатека с правильными именами**.
Любая правка оценивается по тому, помогает ли она этой цели.

## Семь принципов

### 1. Push-only от бэкенда

Фронт обращается к серверу **только** в ответ на действие пользователя.
Никаких `setInterval`, `setTimeout`-поллингов, авто-обновлений по таймеру
в браузере.

**Как применять:**
- Любое предложение «опрашивать сервер каждые N секунд» с фронта —
  отклонять. Решение всегда — push с бэкенда через SSE.
- Если фронту нужно поддерживать какое-то «живое» состояние (просмотр,
  редактирование) — это решается серверным TTL/таймером, а не клиентским
  пингом.
- Исключение — агенты на бэкенде; они работают по своим расписаниям и
  это часть их природы.

### 2. Бэкенд — единственный инициатор обновления UI

Изменения состояния, прогресса, очередей публикуются агентами в SSE.
Фронт перерисовывается на основании этих сообщений и **не делает
дополнительных GET-запросов** для добивания недостающих данных после
SSE-события.

**Как применять:**
- SSE-событие должно содержать всё, что нужно фронту, — иначе
  расширяем payload, а не делаем follow-up fetch.
- Heartbeat'ы ради «оживления UI» (только пульсация без данных) — анти-
  паттерн; визуальная активность выводится из реальных событий.
- При рефакторинге SSE целимся в **дельту** изменений, а не полный
  объект сущности.

### 3. Глубокая целостность данных и crash-tolerance

Любой процесс обязан переживать падение в любой точке и автоматически
продолжаться с места обрыва. БД — единственный источник истины.

**Как применять:**
- Любая новая операция в агентах/конвейере проектируется как ресьюмабл:
  состояние сохраняется в БД до начала фазы, фаза идемпотентна.
- Никаких «состояний только в памяти» (кроме строго эфемерных).
- Двойственное хранение одного и того же факта (например,
  `Series.state` + `SeriesStatus`) — потенциальная рассинхронизация;
  при изменениях в этой зоне обязательно проверять, что оба места
  обновляются атомарно, либо устранять дублирование.
- Обновления через SSE могут теряться (queue.Full) — это допустимо
  **только если** состояние всегда восстанавливается из БД при
  следующем событии или загрузке.

### 4. Дизайн — неприкосновенный контракт

Каждый отступ, угол, размер, цвет, длительность анимации, тип кнопки
выверены и продуманы. Любая оптимизация UI **обязана** сохранять
визуальное поведение «до пикселя».

**Как применять:**
- Изменения CSS, шаблонов, transition'ов, классов карточек, модалок,
  отступов — **только по явной просьбе пользователя**. Performance-
  рефакторинг фронта не оправдывает изменение визуала.
- Если оптимизация требует визуальной корректировки (например,
  заменить `transition: width` на `transform: scale`) — обязательно
  согласовать **до** правки, явно описав, что именно поменяется в
  поведении/виде.
- При рефакторинге Vue-компонентов структура DOM, классы и атрибуты
  должны оставаться идентичными, чтобы CSS продолжал применяться без
  правок.
- При сомнении «безопасно ли это для дизайна» — спрашивать.

### 5. Мгновенный отклик на действие пользователя

На любое нажатие интерфейс отвечает в тот же кадр: спиннер, скелетон,
оптимистическое изменение, предпросмотр, disabled-состояние с явной
визуальной меткой — что угодно, но не пустая задержка.

**Как применять:**
- Сетевая операция не блокирует визуальный feedback. Сначала
  перерисовать UI, потом отправлять запрос.
- Эталон — `toggleAutoScan` (оптимистическое обновление + откат при
  ошибке): новые действия по возможности проектировать так же.
- Кнопки с долгой операцией должны менять вид (спиннер/иконка
  загрузки), а не просто `disabled`.
- Модалки, открывающиеся с подгрузкой данных, показывают скелетон, а
  не пустоту.

### 6. Категоризированное логирование с регулируемыми уровнями

Каждый процесс пишет в свою группу с настраиваемым уровнем детализации.
Логи — диагностический инструмент, а не побочный эффект.

**Как применять:**
- Любой новый процесс/агент/парсер заводит свою группу логирования,
  не пишет в чужую.
- Уровень логирования настраивается через существующую систему
  (`logger.py` + `debug_manager.py`); не хардкодить «всегда verbose».
- Добавляя `logger.info(...)` — указывать осмысленную группу, а не
  общую «app»/«main».

### 7. Единый источник правды

Один факт хранится строго в **одном** месте. Денормализация запрещена;
если нужна сводка/агрегат — это вычисляемое значение, а не отдельная
колонка, файл или кэш, существующий параллельно с источником.

**Как применять:**
- Перед добавлением нового поля/таблицы/кэша спрашивать: «не дублирует
  ли это уже существующий факт?». Если дублирует — оформлять как
  вычисляемое значение, а не как параллельное хранилище.
- Кэши допустимы только при наличии явного механизма инвалидации; они
  не используются как источник для логики (только для быстрого
  чтения).
- При обнаружении существующей двойственности (`Series.state` +
  `SeriesStatus`, и подобных) — **сначала** трассировать все
  зависимости, понять, почему дубль был сделан исторически, и
  **только потом** аккуратно устранять. Двойственность — почти всегда
  симптом важной зависимости, которую нельзя разрушить случайно.
- Дублирование «на всякий случай» / «для удобства» — анти-паттерн.

## Дисциплина внесения изменений

Совпадает с глобальным правилом пользователя:

| Категория                                                                       | Подтверждение |
|---------------------------------------------------------------------------------|---------------|
| Архитектурные правки (новые модули, смена API, разбиение `db.py`)               | **обязательно** |
| Функциональные правки (логика агентов, шагов конвейера, рендера карточек)       | **обязательно** |
| Правки CSS/HTML/визуала (любые)                                                  | **обязательно** (см. принцип 4) |
| Создание правил, конфигов, документации по запросу                              | без подтверждения |
| Чтение, поиск, обзор архитектуры, планирование                                  | без подтверждения |
| Запуск тестов, линтеров, статических проверок                                   | без подтверждения |

Алгоритм:

1. Прочитать код, изучить контекст.
2. Сформулировать план: файлы, правки, риски, как принципы не нарушаются.
3. Показать план.
4. Дождаться явного «да/делай/применяй».
5. Только после этого править.

## Дисциплина находок

При обнаружении мелких дефектов (дубли методов, мёртвый код, опечатки в
именах, рассинхронизированные комментарии, странные конструкции) **в
ходе** рефакторинга или иной работы:

1. Не молчать, не откладывать «в потом».
2. Сообщить пользователю: что найдено, в каком файле, что это с точки
   зрения архитектуры (роль метода/блока), почему вызывает вопросы.
3. Принять решение **на месте**: чинить здесь же, выписать в отдельный
   список, или осознанно оставить.
4. Не накапливать «технический долг молча» — каждое найденное
   несоответствие либо устраняется, либо документируется явно.

## Тестирование и среда

Текущая среда — продакшн-однопользовательский инстанс. На него завязаны:
- реальный qBittorrent с десятками активных торрентов;
- реальная медиатека на диске, на которую ссылается БД;
- реальные авторизации в трекерах (Kinozal, RuTracker, qBit);
- реальные TMDB-привязки.

Любая правка пайплайна (скан, загрузка, нейминг, перемещение, нарезка)
способна повредить эти данные так, что восстановление займёт дни. До
появления отдельного тестового стенда (отдельный qBittorrent + отдельный
файловый каталог + муляжи трекеров) рефакторинг бэкенда выполняется:

- с обязательным резервным копированием БД и медиатеки **перед** любым
  применением изменений;
- с разделением на минимальные обратимые шаги;
- с дополнительной верификацией поведения после каждого шага;
- с готовностью откатить шаг, если поведение хоть в чём-то отклонилось.

## Команды разработки

Виртуальное окружение — в `venv/` в корне репозитория (не в git).

| Задача | Команда |
|---|---|
| Активация окружения | `source venv/bin/activate` |
| Штатный запуск | `gunicorn -c gunicorn_config.py run:app` (порт 5000) |
| Запуск как сервис | `sudo systemctl start/stop/status series-tracker.service` |
| Зависимости | `pip install -r requirements.txt`, затем `playwright install` |
| Логи приложения | `logs/*.log` — ротируемые JSON-файлы |

Неочевидности запуска:

- **Фоновые агенты стартуют только под Gunicorn** — в хуке `post_fork`
  (`gunicorn_config.py` → `run.post_fork_hook`). Прямой запуск
  `python run.py` поднимет Flask **без агентов**: годится для проверки
  API/фронта, но конвейер (скан/загрузка/нарезка/переименование)
  работать не будет.
- **`workers = 1` в Gunicorn — инвариант, а не настройка
  производительности**: агенты — потоки внутри воркера; второй воркер
  продублировал бы их и устроил гонки за qBittorrent, БД и файлы.
  Параллелизм обеспечивается `threads = 10` (`gthread`).
- `reload = True` в конфиге: воркер перезапускается при изменении
  файлов — вместе со всеми агентами.
- БД — SQLite-файл `app.db` в корне репозитория. Перед любыми правками
  пайплайна — резервная копия (см. «Тестирование и среда»).
- **Тестового фреймворка нет.** `remote_test_runner.py`,
  `tmdb_test_script.py`, `Downlodaer_test/` — ad-hoc-скрипты,
  `routes/test.py` — тестовые эндпоинты. Проверка изменений — ручная
  верификация на живом инстансе по правилам раздела
  «Тестирование и среда».
- Внешние бинарники, которые ищет код (`utils/path_finder.py`):
  `ffmpeg` (нарезка), `yt-dlp` (загрузка VK).

## Модель выполнения и конвейер

Путь серии от источника до медиатеки (каждый шаг — ресьюмабл,
состояние фаз — в БД, прогресс публикуется через `StatusManager` →
`sse.py` → фронт):

```text
MonitoringAgent (по расписанию) ──► scanner.perform_series_scan()
    │   парсер трекера (parsers/) или scrapers/vk_scraper.py (Playwright)
    │   RuleEngine (профиль правил) ─ извлекает сезон/серию/озвучку/качество
    │   для VK: SmartCollector строит план загрузки (одиночные + компиляции)
    ▼
Agent ──► торренты в qBittorrent (qbittorrent.py)
DownloaderAgent ──► VK-видео через yt-dlp (downloader.py)
    ▼
SlicingAgent ──► компиляции режутся ffmpeg по главам (utils/chapter_*)
    ▼
RenamingAgent ──► logic/renaming_processor + filename_formatter
    ▼
файл со строгим именем в медиатеке (итоговый артефакт)
```

Опорные точки, которые не видны из одного файла:

- `db.py` — единый класс `Database` со **всеми** CRUD-операциями
  (~1800 строк); агенты и роуты ходят в БД только через него.
  Его разбиение — предмет ТЗ-1 рефакторинга, без подтверждения
  не трогать.
- Профили правил (`rule_engine.py`) — общая механика и для VK-названий,
  и для имён файлов в торрентах; редактируются визуальным конструктором
  на фронте (Настройки → Фильтры VK).
- `auth.py` (`AuthManager`) — единая точка сессий Kinozal, RuTracker,
  qBittorrent; парсеры сами авторизацию не ведут.
- Один HTTP-эндпоинт SSE (`routes/system.py`) обслуживает все
  push-обновления; фронт держит одно соединение.

## Язык и стиль

- Все ответы, объяснения, комментарии в коде — **на русском**.
- Тон: профессиональный инженерный, «партнёр по кодингу».
- Перед отправкой ответа проверять, что текст не «съехал» на английский,
  особенно в длинных ответах с примерами кода.
