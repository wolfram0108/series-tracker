**Это переписывание с нуля, а не копирование и не перенос.**

1. **Ни одной строки бездумного переноса.** Каждый аспект старой
   системы проходит цикл: разобрать, как сделано → понять, **зачем**
   (какую реальную задачу решает) → решить, **верно ли это вообще**
   и оптимально ли → спроектировать своё решение → реализовать с нуля.
   «Старый подход правильный» — допустимый **вывод** разбора и
   запрещённое **умолчание**.
2. **Никаких чёрных ящиков.** Непонятый кусок не «аккуратно
   сохраняется» — он разбирается до конца либо выносится вопросом
   пользователю. Непонятое не переносится.
3. **Старый код в `main` — справочник требований и улики о граблях**
   (см. [contracts/findings.md](contracts/findings.md)), не образец.
4. **Правильность нового кода** доказывается его собственными тестами
   и чек-листом сценариев — не дифом со старым поведением.
5. **Форма работы:** перед реализацией модуля — короткий разбор
   пользователю («как сейчас, зачем на самом деле, что неверно, как
   предлагаю»); ключевые решения принимаются вместе, по месту, и
   фиксируются в [contracts/revision.md](contracts/revision.md).

## Архитектура (кратко; подробно — в ТЗ)

Один asyncio-процесс: «тупая» шина (`core/bus.py`, только
publish/subscribe + wildcards) и модули (`modules/*`), общающиеся
конвертами (`core/envelope.py`: event / command / query / reply).
Request/reply реализован в обвязке `core/module.py` ПОВЕРХ pub/sub —
шина о нём не знает и логики не содержит (чек-лист «чего шина не
делает» — в ТЗ, раздел 2.2). Gateway (`modules/gateway`) — FastAPI,
единственная HTTP/SSE-точка; событие шины → SSE по таблице `SSE_MAP`.

Правила модулей:
- модуль импортирует только `core/*`; импорт из чужого модуля запрещён;
- **БД — приватная память модуля, не участник шины** (решение Р-7):
  каждая таблица/колонка принадлежит ровно одному модулю (карта — в ТЗ,
  раздел 6); чужие данные — только query через шину; свой SQL —
  в `repository.py` модуля; значимые изменения данных публикуются
  событием;
- **«шина — сигнал, БД — истина»**: задача персистится до публикации
  события; при старте модуль добирает pending из БД (хук `on_start`);
- блокирующие операции — только в executor'ах с таймаутами (зависание
  одного модуля не должно замораживать процесс — находка №7);
- группа логирования = имя модуля (`core/logging.py`).

## Фронт

`static/` и `templates/` принесены из `main` как есть. Визуал, DOM,
классы, анимации — **неприкосновенны**. JS-слой данных (fetch,
обработчики SSE) адаптируется под новый контракт — каждая правка
согласуется с пользователем до внесения.
