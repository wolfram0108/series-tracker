## 7. Хронические зависания воркера: ~190 WORKER TIMEOUT за полгода
(Первая версия этой находки ошибочно винила `max_requests=1000` —
опровергнута журналом systemd.) Журнал юнита wolfram-ts.service
(мастер pid 245): с декабря 2025 по июнь 2026 — около 190 событий
`[CRITICAL] WORKER TIMEOUT`. Воркер зависает намертво (все потоки
разом, SSE обрывается посреди строки), мастер ждёт `timeout = 120`
секунд, убивает его и пересоздаёт — с полным рестартом агентов и
переавторизацией qBittorrent; итоговое окно недоступности ~2 минуты.
Зависания идут сериями с интервалом ~1,5 часа, в том числе ночью без
какой-либо нагрузки (12–13 мая — вся ночь подряд): триггер внутренний
(агенты/фоновые циклы), а не HTTP-трафик. «Стабильность» прода — это
watchdog Gunicorn, полгода маскировавший хроническую болезнь.
Первопричину в старом коде не ищем (код пересоздаётся), но:
**Требования к новой системе:** (а) сохранить watchdog уровня процесса
(systemd Restart=always + healthcheck); (б) перезапуск без 2-минутного
окна — быстрый старт модулей, reconcile вместо «первоначальных
проверок»; (в) зависание одного модуля не должно замораживать весь
процесс — блокирующие операции только в executor'ах с таймаутами;
(г) на этапе 4 при переносе агентов выяснить, что в них тикает с
периодом ~1,5 часа — кандидат в первопричину, должен быть учтён.

## 8. Транзиентные 403 от qBittorrent
2026-06-11 11:23 — серия «Не удалось получить список файлов», затем
«Получен статус 403 (Forbidden). Попытка повторной аутентификации» —
сессия qBit протухает, клиент переавторизуется, но успевает насыпать
ERROR-ов и, возможно, сорвать операции (renaming_processor предупреждал
об отсутствии файлов). **Требование:** модуль torrents переавторизуется
тихо и повторяет исходную операцию, не роняя задачу.

## 9. Фиктивная проверка безопасности в файлбраузере
routes/filebrowser.py: список allowed_roots = ['/', '/nas', '/home',
'/tmp'] — наличие '/' разрешает ЛЮБОЙ абсолютный путь, проверка не
делает ничего. В новой системе (modules/library) иллюзия убрана:
корни — явная настройка конструктора, по умолчанию вся ФС (фактическое
старое поведение в однопользовательском LAN-сервисе), нормализация
пути до проверок.

## 10. TMDB-клиент глотает ошибки
utils/tmdb_client.py возвращает []/None при любой ошибке (сеть, токен,
404) — проблема тонет, пользователь видит «ничего не нашлось» вместо
причины. Новый modules/metadata пробрасывает ошибку в reply.

## 11. RuTracker-парсер гадает трёхуровневым каскадом
rutracker_parser.py: поиск даты — таблица → любые <li> → «любая похожая
на дату строка где угодно, если в 100 символах рядом есть „дата“, „reg“
или „в“» (слово «в» есть в любом русском тексте). Аналогично поиск
ссылки. Новый парсер — детерминированные селекторы (h1.maintitle,
table.attach «Зарегистрирован», a.dl-stub) с громкой ошибкой при
ненахождении (включая случай гостевой сессии — покрыт тестом).

## 12. Браузер там, где есть API
anilibria_parser.py рендерил Vue-фронтенд Анилибрии головным Firefox,

## 13. Ловушка форматов дат при переключении
torrents.date_time хранит строки в формате СВОЕГО трекера (kinozal
'DD.MM.YYYY HH:MM:SS', anilibria ISO-Z, astar 'DD.MM.YYYY'); механизм
«обновилось/нет» — строковое сравнение. Унификация формата в новой
системе вызвала бы ложное «всё обновилось» при переключении и массовую
перекачку. Решение: маркеры сохраняют исторический формат по трекерам
(modules/sources/dates.py).

## 24. SSE series_updated несёт неполный объект сериала
_update_and_broadcast шлёт голую строку таблицы series: без statuses
(фронт парсит строку state.split(',')), без is_busy / tmdb_info /
downloaded_episodes_count, которые есть в GET /api/series. Поэтому
другие события (relocation_finished, renaming_complete) компенсируют
это полным loadInitialSeries — follow-up fetch, анти-паттерн по
принципу 2. Во фронте при этом уже заложена мёртвая ветка
`s.statuses || s.state.split(...)` (app.js:66) — фронт готов принять
массив статусов, бэкенд его никогда не присылал.

## 25. Каскадные дубли broadcast при одном изменении
status_manager: каждый set_status() сам вызывает _update_and_broadcast;
sync_torrent_statuses вызывает set_status дважды (downloading, ready) →
2 события series_updated на сериал за такт; старт скана — тоже 2
(error=False, scanning=True). Вместе с тактом 5 с в MonitoringAgent
(_update_active_statuses по ВСЕМ сериалам независимо от изменений) —
это и есть механика SSE-шторма из находки 4.

## 26. Ресьюмабельность scan_tasks — фиктивная
scanner.py принимает recovery_mode/existing_task для продолжения
прерванной задачи, но НИКТО не вызывает его с recovery_mode=True
(grep по репо пуст). «Восстановление» при старте
(Agent._recover_scan_tasks_from_db) просто УДАЛЯЕТ незавершённые
scan_tasks с комментарием «чтобы избежать бесконечных циклов при
ошибках». Итог: упавший посреди добавления торрентов скан не
продолжается — задача тихо выбрасывается. Механика журнала
(task_data + results_data по индексам) при этом рабочая.

## 27. Скан без блокировок: ручной и автоматический могут пересечься
POST /api/series/<id>/scan выполняет perform_series_scan синхронно в
HTTP-запросе (для VK — минуты: API-пагинация + правила + план; фронт
ждёт fetch всё это время). Автоскан-поток (_perform_full_scan)
пропускает сериал лишь при наличии agent_tasks — взаимной блокировки
с ручным сканом того же сериала нет, два скана могут идти параллельно
и наперегонки писать торренты/медиа-элементы.

## 28. Тип замены трекера захардкожен подстрокой
scanner.py:343: site_type = 'fixed' if 'astar' in canonical_name else
'rolling' — типизация трекера зашита строкой в оркестраторе. rolling-
замена срабатывает только когда активный торрент ровно один; при 2+
активных новый торрент добавляется без замены (вероятно, осознанное
«не знаем, что заменять — не заменяем», но нигде не сказано).

в monitoring_agent (копипаста). Симптомы редактирования «вслепую».
