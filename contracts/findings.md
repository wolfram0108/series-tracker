## 7. Хронические зависания воркера: ~190 WORKER TIMEOUT за полгода
(Первая версия этой находки ошибочно винила `max_requests=1000` —
опровергнута журналом systemd.) Журнал юнита wolfram-ts.service
(мастер pid 245): с декабря 2025 по июнь 2026 — около 190 событий
`[CRITICAL] WORKER TIMEOUT`. Воркер зависает намертво (все потоки
разом, SSE обрывается посреди строки), мастер ждёт `timeout = 120`
секунд, убивает его и пересоздаёт — с полным рестартом агентов и
переавторизацией qBittorrent; итоговое окно недоступности ~2 минуты.
Зависания идут сериями с интервалом ~1,5 часа, в том числе ночью без
какой-либо нагрузки (12–13 мая — вся ночь подряд): триггер внутренний
(агенты/фоновые циклы), а не HTTP-трафик. «Стабильность» прода — это
watchdog Gunicorn, полгода маскировавший хроническую болезнь.
Первопричину в старом коде не ищем (код пересоздаётся), но:
**Требования к новой системе:** (а) сохранить watchdog уровня процесса
(systemd Restart=always + healthcheck); (б) перезапуск без 2-минутного
окна — быстрый старт модулей, reconcile вместо «первоначальных
проверок»; (в) зависание одного модуля не должно замораживать весь
процесс — блокирующие операции только в executor'ах с таймаутами;
(г) на этапе 4 при переносе агентов выяснить, что в них тикает с
периодом ~1,5 часа — кандидат в первопричину, должен быть учтён.

## 24. SSE series_updated несёт неполный объект сериала
_update_and_broadcast шлёт голую строку таблицы series: без statuses
(фронт парсит строку state.split(',')), без is_busy / tmdb_info /

это полным loadInitialSeries — follow-up fetch, анти-паттерн по
принципу 2. Во фронте при этом уже заложена мёртвая ветка
`s.statuses || s.state.split(...)` (app.js:66) — фронт готов принять
массив статусов, бэкенд его никогда не присылал.

## 25. Каскадные дубли broadcast при одном изменении

## 26. Ресьюмабельность scan_tasks — фиктивная
scanner.py принимает recovery_mode/existing_task для продолжения
прерванной задачи, но НИКТО не вызывает его с recovery_mode=True
(grep по репо пуст). «Восстановление» при старте
(Agent._recover_scan_tasks_from_db) просто УДАЛЯЕТ незавершённые

ошибках». Итог: упавший посреди добавления торрентов скан не
продолжается — задача тихо выбрасывается. Механика журнала
(task_data + results_data по индексам) при этом рабочая.

## 27. Скан без блокировок: ручной и автоматический могут пересечься
POST /api/series/<id>/scan выполняет perform_series_scan синхронно в
HTTP-запросе (для VK — минуты: API-пагинация + правила + план; фронт
ждёт fetch всё это время). Автоскан-поток (_perform_full_scan)
пропускает сериал лишь при наличии agent_tasks — взаимной блокировки
с ручным сканом того же сериала нет, два скана могут идти параллельно
и наперегонки писать торренты/медиа-элементы.

## 30. Очередь загрузок: полный список в SSE каждые 5 секунд
DownloaderAgent._tick шлёт download_queue_update со ВСЕМИ активными
задачами каждые 5 с независимо от изменений + agent_heartbeat — вторая
половина SSE-шторма (к находкам 4/25). Прогресс в БД при этом пишется

## 31. Ошибочные загрузки ретраятся только перезапуском сервиса
requeue_stuck_downloads при старте переводит в pending задачи и
'downloading', и 'error'. В работающем процессе error-задачи не
ретраит никто, лимита попыток нет (attempts копится в никуда). При
следующем скане get_download_task_by_uid ищет только
pending/downloading → для того же uid создаётся ВТОРАЯ задача, старая
error-запись остаётся мусором до ручной очистки очереди.

## 32. Задачи, выпавшие из плана, зависают в pending навсегда
Добор задач JOIN'ится по plan_status ∈ in_plan_* — если элемент после
пересчёта плана выпал (replaced/redundant), его pending-задача больше
никогда не будет выбрана и не удаляется. Копится до ручного clear.

## 33. Задачи single_vk создаются и удаляются без выполнения
logic/task_creator.py при каждом скане создаёт renaming_tasks типа
'single_vk' (с old_path/new_path), но диспетчер RenamingAgent
(_process_task) обрабатывает только mass_torrent_reprocess и
mass_vk_reprocess — для single_vk нет ветки, задача помечается
in_progress и тут же удаляется как «успешно завершённая», не сделав
ничего. Реальное переименование одиночных файлов выполняет
mass_vk_reprocess при каждом скане. Побочный эффект: пока single_vk
задачи висят pending, сериал считается is_busy (блокируется карточка).

## 34. get_pending_renaming_task определён в Database ТРИЖДЫ
db.py:1326, 1587, 1740 — три версии метода с разными сигнатурами и

## 35. Автофильтр глав всегда бракует главу на 00:00:00
ChapterFilter._estimate_chapter_duration «оценивает длительность» по
времени НАЧАЛА главы (время конца недоступно), поэтому глава, начинающаяся
в 00:00:00, всегда получает «слишком короткая: 0сек» и бракуется. На
практике первая глава компиляции — почти всегда опенинг, так что
эвристика «угадывала» по ложной причине. Порт 1:1 (контракт UI —
согласовано в Р-16); пересмотр эвристик — отдельным решением, если
когда-нибудь понадобится.

## 36. Ошибка переобработки блокировала карточку навечно

активная работа (Р-17, согласовано); ошибка остаётся носителем в
задаче и перевыполняется при следующем скане/старте.

## 38. series_updated обязан нести is_busy в каждом событии

## 39. Очередь торрент-агента: HTTP и SSE отдавали разные формы

строк были undefined до первого SSE. В новой системе обе выборки идут

## 41. Булевы колонки: SQLite 0/1 против true/false старого контракта
Обнаружено пользователем на стенде (этап 6): данные конструктора
перенеслись целиком, но переключатель «продолжить обработку» у всех
правил выглядел выключенным. Это не повреждение JSON — недоконтракт:
старый бэкенд (SQLAlchemy Boolean) отдавал true/false, новый (sqlite3
напрямую) — 0/1, а v-model чекбокса в Vue 3 сравнивает строго и не
считает 1 включённым. Затронуты все BOOLEAN-колонки, уходящие наружу:
auto_scan_enabled (series), continue_after_match (parser_rules),
is_ignored_by_user / is_available (media_items), is_active (torrents).
Фикс: Database.coerce_bools — владельцы таблиц приводят свои булевы
колонки на выдаче (catalog, rules, scan, torrents).

## 42. Детектор «разлогинены» Kinozal не узнавал форму со слешем
Обнаружено пользователем на стенде (этап 6): добавление сериала Kinozal
давало 400, в логе «ссылка download.php не найдена». Мониторинг
(категоризированные логи новой системы, group=sources) сразу показал
причину: trackerauth отдавал sources гостевую страницу без релогина.
KinozalProvider.is_logged_out искал точную подстроку
'action="takelogin.php"', а реальная форма входа — action="/takelogin.php"
(со слешем). Детектор возвращал False для гостя → релогин не
запускался → парсер не находил ссылку. Старая система проблемы не имела
по другой причине: она логинилась при КАЖДОМ создании сессии (кэш
только в памяти), без детектора. Новая — лениво (персистентные куки,
переживает рестарт), поэтому детектор обязан быть точным. Фикс: regex
action=['"]?/?takelogin\.php (устойчив к слешу и кавычкам); на
залогиненных страницах формы нет вовсе. Регрессия — в test_trackerauth.

## 43. Торрент в qBit-state error/missingFiles показывался как «загрузка»
Обнаружено пользователем на стенде (этап 6): добавил торрент с неверным
путём, qBittorrent держит раздачу в state="error", но карточка показывала
«Загрузка». Причина — свёртка torrents._contribute: qBit-состояния
ошибок не входили в _NOT_DOWNLOADING, прогресс<100 → флаг downloading.
Тот же дефект был в старой системе (status_manager.sync_torrent_statuses:
status not in ['pausedUP','pausedDL','uploading']) — унаследован, не

## 45. Не было штатной отмены активной загрузки VK (по просьбе пользователя)

## 46. Многопоточная загрузка VK (hls+-N) + мониторинг постобработки
По ТЗ пользователя. Замер на стенде: прирост скорости даёт ТОЛЬКО
связка hls-формат + yt-dlp -N (×31: 55МБ→1740МБ за 25с, насыщает канал
500 Мбит); -N без hls не помогает (best/DASH — без эффекта,
подтверждено). Реализация двухфазная, обе фазы мониторятся ИЗ ВЫВОДА
инструмента (мониторинг по размеру файла на диске запрещён —
требование пользователя):
- фаза download: yt-dlp -f "b[protocol*=m3u8]/b" -N <threads>
  --fixup never (постобработку yt-dlp отключаем, делаем сами);
- фаза remux: наш ffmpeg -c copy -progress pipe:1 — прогресс и ETA из
  метрик ffmpeg (out_time/длительность, speed), статус задачи
  'processing'. Постобработка обязана показывать прогресс и оставшееся
  время (ТЗ) — закрыто слепое пятно «100% загрузки, файла нет».
Настройка yt_dlp_concurrent_fragments (дефолт 6, насыщает 500 Мбит),
поле в UI рядом с параллельными загрузками. requeue при старте
возвращает и processing. Отмена (находка 45) работает на обеих фазах.

## 47. Парсер прогресса не понимал формат hlsnative+-N
Обнаружено пользователем: при новой загрузке статус «загрузка», но
прогресс/скорость/ETA пустые. Причина: формат строки yt-dlp при
hlsnative+-N отличается от обычного —
"[download]   3.0% of ~  24.82GiB at 5.00MiB/s ETA Unknown (frag 138/4602)":
пробелы после '~', ETA='Unknown', хвост '(frag X/Y)'. Старый
PROGRESS_RE (под формат progressive, '~' вплотную, ETA только время)
не матчил ни одной строки → ни одного события прогресса. Исправлен
regex (\s* после ~, ETA '[\d:]+|Unknown'). Корневая причина пропуска
тестами — находка 47-методология (см. ниже): тест парсера был написан
на ВЫДУМАННОЙ строке, а не на реальном выводе инструмента, поэтому
кодировал ту же ошибку, что и regex. Добавлен тест на реальную
hlsnative-строку, захваченную со стенда.
