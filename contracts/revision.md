# Журнал решений ревизии

Каждое решение — после разбора «как сейчас → зачем → верно ли», с
согласованием пользователем. Формат: контекст, решение, обоснование.

---

## Этап 2: trackerauth + torrents (решения от 2026-06-11)

### Р-1. Границы: скачивание .torrent уходит из qBit-клиента
**Было:** `qbittorrent.py:add_torrent()` сам скачивал .torrent с
трекеров, зная их причуды (Referer Кинозала, заголовки RuTracker).
**Решено:** `torrents` говорит только с qBittorrent API; скачивание
.torrent и кэш `torrent_cache/` — у `sources`; авторизованные
запросы к трекерам — query `trackerauth.fetch`. Трекеро-специфика
живёт у владельцев трекеров. *(Согласовано пользователем.)*

### Р-2. Hash торрента вычисляется локально
**Было:** временный тег + поллинг qBit до 15×2 с, теговые баги,
неразличимый «Fails.».
**Решено:** infohash v1 считается из .torrent-файла (SHA1 байтов
секции `info`), для magnet берётся из `btih` (hex и base32);
`btmh`/чистый v2 — явная ошибка «не поддерживается», не молчание.
Страховки: юнит-тест на реальных файлах (hash обязан совпасть с
qBittorrent), контрольный запрос по hash после добавления.
*(Согласовано: «всё звучит отлично».)*

### Р-3. Сессия qBittorrent эфемерна, SID в БД не хранится
**Было:** SID сохранялся в `settings`; при старте почти всегда
протухший → 403-шум (находка №8), дубль чужого состояния в БД.
**Решено:** логин при старте (~100 мс, локальный сервис без лимитов);
на 403 посреди работы — тихий релогин и повтор сорванной операции,
ошибка наверх только при повторной неудаче. *(Согласовано.)*

### Р-4. Сессии трекеров — наоборот, персистентны
**Было:** куки трекеров жили только в памяти → каждый рестарт = новые
логины (при хронических рестартах прода — сотни логинов на Кинозал);
живость сессии никогда не проверялась — протухание ломало парсинг
далеко от причины.
**Решено:** куки хранятся в БД (таблица `tracker_sessions`, владелец
trackerauth; добавляется Alembic-миграцией поверх базовой схемы);
при первом использовании после рестарта — дешёвая проверка живости;
детект протухания посреди работы → релогин → повтор запроса (один
раз); логины с ограничением частоты на трекер.
**Мотив пользователя (ценность проекта):** частые логины и лишние
запросы вредят чужому сайту — «мы хотим уважать других». Это правило
распространяется на все внешние обращения системы (парсеры, сканер).
*(Согласовано.)*

### Р-5. HTTP-стек
**Решено:** httpx (async) — для qBittorrent API; requests +
cloudscraper в executor с жёсткими таймаутами — для трекеров
(cloudscraper необходим для Astar за Cloudflare и работает только
поверх requests). Любой сетевой вызов обязан иметь таймаут —
зависание одного модуля не должно замораживать процесс (находка №7).
*(Согласовано после разъяснения sync/async/executor.)*

### Р-6. Девять мёртвых таблиц не входят в базовую схему
**Контекст:** находка №1 — в прод-БД 29 таблиц, в коде используются 20.
**Решено:** Alembic-миграция №1 (базовая схема новой системы) включает
только живые таблицы; при финальном переключении лишние таблицы в
копии прод-БД не мешают и чистятся отдельной миграцией после
стабилизации.

### Вне ревизии (констатации)
- Стадия `awaiting_metadata` для magnet — физика протокола (анкета
  доезжает от пиров позже hash), сохраняется в новом `torrents`.
- Детект HTML-ответов при скачивании .torrent («суточный лимит»,
  «включите JavaScript») — требование к `sources` (этап 3).
- Long-poll `sync/maindata`: таймаут — норма, не ретраить.
- 404 от qBit = ресурса нет, не ретраить.

### Р-7. БД — приватное хранилище модуля, не участник шины
**Вопрос пользователя:** «модуль базы данных — такой же модуль на
шине?» **Решено (после разбора):** нет. БД — приватная память модуля
(состояние между перезапусками), межмодульное общение — только шина.
Дисциплина, делающая это безопасным:
- каждая таблица принадлежит ровно одному модулю (карта — ТЗ, раздел 6);
  чужие данные — только query через шину; канал к чужим данным мимо
  шины запрещён;
- SQL инкапсулирован в `repository.py` модуля, не размазан по коду;
- значимые изменения данных модуль публикует событием в шину —
  «что меняется» видно на шине, «как лежит» — приватно;
- обращения к БД логируются группой модуля (принцип 6).
**Отвергнутая альтернатива** (БД-модуль на шине): возвращает god-object
уровня старого db.py, ломает транзакции (конверт не переносит открытую
транзакцию), на порядки дороже на каждый доступ к собственным данным.
*(Согласовано пользователем.)*

## Этап 3: движок правил (решения от 2026-06-11)

### Р-8. Семантика нового движка правил
Формат блочных правил в БД — констрейнт (профили пользователя работают
как есть). Семантика повторяет фактическую, кроме трёх согласованных
исправлений:
- **(А, баг — чинить)** несколько извлекающих действий в одном правиле:
  у каждого действия свой regex и СВОЯ нумерация capture-групп; старый
  сквозной счётчик молча терял все извлечения после первого успешного
  (подтверждено на 5 правилах в 3 профилях; у «AniDub VK Rules»
  терялся сезон).
- **(Б)** битое/пустое условие — громкая ошибка валидации профиля:
  правило помечается неисправным и не применяется, в лог — что именно
  сломано; старое поведение (условие тихо исчезает, правило ширеет) —
  запрещено.
- **(В)** остановка обхода (continue_after_match=0) — только если
  правило ДАЛО результат (извлекло данные или exclude); «условия
  совпали, но шаблон извлечения нет» больше не глушит каскад. На
  текущих профилях пользователя поведение не меняется (в их
  стоп-правилах условие = шаблону извлечения); пользователь: «то, что
  было, — скорее всего борьба с чем-то другим, поэтому так нелогично».
Сохраняется как есть: OR-группировка условий (оператор связывает со
следующим, группы И через ИЛИ), правило «отрицательный результат
add/subtract игнорируется» (именное правило пользователя),
регистронезависимость, перезапись полей поздними правилами, ключи
extracted (episode/season/start/end/voiceover/quality/resolution).
Новое: компиляция профиля один раз с кэшем (вместо рекомпиляции на
каждое название).
**Верификация:** дифф старого и нового движков на ~1100 реальных
названиях из фикстуры (350 VK + 737 файлов торрентов), каждое
расхождение разбирается поимённо. *(Согласовано.)*

### Р-9. Модуль sources: факты вместо решений, API вместо браузера
- Контракт `sources.parse` → {title, releases[]}: парсер отдаёт ФАКТЫ
  (ссылка/магнет, маркер даты, серии, качество); решение «ново/старо»
  принимает scan. Старый протокол link=None упразднён.
- Anilibria — через официальный публичный API (находка 12), включая
  готовый infohash. *(Пользователь: «API — лучшее, что можно
  придумать, однозначное согласие».)*
- Маркеры дат — в исторических форматах трекеров (находка 13, защита
  от массовой перекачки при переключении).
- Доставка страниц: kinozal/rutracker — через trackerauth.fetch;
  astar/anilibria_tv — общий Playwright-пул на этапе 6 (находка 14:
  сейчас ими никто не пользуется; функции разбора готовы и
  протестированы по коду).
- Межзапросная пауза на трекер (3 с) + замок «одна операция на трекер»
  («уважать других»).
- Скачивание .torrent (Р-1) — здесь же: кэш torrent_cache/, детект
  HTML-ответов (суточный лимит, JS).

### Р-10. Планировщик покрытия и VK-источник
- Семантика SmartCollector v4.1 перенесена 1:1 как чистая функция
  modules/scan/planner.py, КРОМЕ согласованного **фикса Г (находка
  22)**: лучшая одиночка выбиралась инвертированно (max вместо min
  ключа качества — худшее разрешение / последний в списке приоритетов
  пользователя; этап апгрейда в том же алгоритме считал наоборот).
  Пользователь: «чинить, приоритет за наивысшим качеством единичной
  серии». Дифф со старым кодом: 9/10 реальных сериалов идентичны,
  1 согласованное отклонение (сериал 87: выбран 2160p вместо 1080p).
- VK — источник в sources через официальный VK API (находка 15):
  токен не покидает trackerauth (провайдер vk подмешивает его в
  параметры), таймауты на всех вызовах (находка 17 — кандидат в
  первопричину зависаний прода), пауза между страницами пагинации,
  предохранитель на 50 страниц.
- **Формулы идентификаторов — констрейнт данных** (существующие записи
  обязаны совпадать): unique_id медиа-элемента =
  md5(url + pub_date.strftime('%Y-%m-%d %H:%M:%S') + series_id)[:16];
  torrent_id = md5(link + date_time)[:16]. Применяются в scan (этап 4).
- Статусы плана: 'replaced' = эпизоды накрыты планом, 'redundant' =
  не накрыты (семантика оригинала сохранена).
- Оркестратор сканирования (цепочка сканов, зеркала-фоллбэк,
  scan_tasks-ресьюмабельность, fixed/rolling замены) — этап 4, вместе
  с конвейером, который он запускает.

## Этап 4: конвейер (решения от 2026-06-12)

### Р-11. Статусная модель: вычисляемая, на свёртках, без хранения
Разбор показал три этажа хранения одного факта (задачи → флаги
series_statuses → строка series.state), поллинг-пересчёт каждые 5 с по
всем сериалам с безусловным broadcast (механика SSE-шторма, находки 4 и
25) и эфемерный по природе viewing в персистентной БД с heartbeat-
таймером на фронте. Согласовано с пользователем:

- **Статус сериала нигде не хранится** — это вычисляемое значение
  (принцип 7). Новая система не пишет и не читает ни series.state, ни
  series_statuses; физически таблица и колонка остаются в схеме
  (инвариант Alembic 0001) до зачистки отдельной миграцией после
  переключения. Пересчёт при старте процесса — приемлем
  («пусть пересчитываются, главное — правильно»).
- **Свёртки вместо опроса**: модуль-владелец задач публикует событие
  `series.status.contribution {source, series_id, flags}` в момент,
  когда его вклад изменился (он сам только что записал это в свою
  таблицу), и при старте после reconcile. Допустимые флаги: scanning,
  metadata, renaming, checking, activating, downloading, slicing,
  ready, error, waiting. Все false = вклад снят.
- **Агрегатор в catalog** (владелец таблицы series): держит вклады в
  памяти (эфемерный кэш, восстанавливается публикациями поставщиков
  при старте), складывает в набор, сортирует по единой иерархии
  (error … waiting — как старый STATUS_HIERARCHY, источник порядка
  пилюль) и публикует `series.status.changed {series_id, statuses}`
  **только при изменении набора**. Пустой набор = ['waiting'].
- **Разная семантика waiting сохранена как осмысленная**: у VK waiting
  приходит в свёртке владельца media_items («есть pending в плане,
  нет активности») и сосуществует с ready (фронт: stripes-stopped);
  у торрентов waiting — только «нет никаких вкладов».
- **viewing — эфемерное состояние в памяти catalog**: команды
  catalog.viewing.start/stop (открытие/закрытие модалки — действия
  пользователя), страховка — событие gateway.sse.clients {count}:
  при count=0 все viewing сбрасываются. Исчезают: DateTime в БД,
  setInterval-heartbeat 30 с на фронте (правка JS — на этапе 5 по
  согласованию), стейл-чекер каждые 5 с. Осознанный компромисс: при
  двух вкладках смерть вкладки с модалкой не сбросит viewing, пока
  жива вторая; усиление (client_id) — на этапе 5, если потребуется.
- **Ошибка получает носителя** (согласовано): задача конвейера при
  ошибке не удаляется, а переходит в stage/status='error' с сообщением;
  ошибка скана — scan_tasks.status='error'. error сериала = «есть
  незакрытая задача-ошибка», сброс — повторным сканом или действием
  пользователя. Это же — честная замена /api/agent/reset (находка 23).
- Контракт с фронтом: массив statuses в payload оживляет мёртвую ветку
  `s.statuses || s.state.split(...)` (находка 24) — JS пилюль/слоёв
  не меняется, визуал идентичен.

### Р-12. Оркестратор сканирования (согласован 2026-06-12)
Разбор: находки 26 (фиктивная ресьюмабельность scan_tasks — recovery
никем не вызывался, незавершённые задачи удалялись), 27 (скан без
блокировок: ручной в HTTP-запросе + автоскан могли пересечься),
28 (тип замены — подстрока 'astar' в оркестраторе), 29 (мусор).
Решения:

- **Зеркала-фоллбэк переехали в sources** (владелец таблицы trackers и
  доставки): sources.parse сам перебирает зеркала, scan о них не знает;
  результат несёт фактический URL.
- **Сравнение «обновилось ли» — по torrent_id** (md5(link+date_time),
  формула-констрейнт Р-10, верифицирована на фикстуре 190/190; для
  anilibria link = magnet — так в прод-данных). Протокол link=None
  старых парсеров не воспроизводится (Р-9).
- **Настоящая ресьюмабельность**: план замен — в scan_tasks ДО
  выполнения; шаги идемпотентны (локальный infohash, повторное
  добавление = existed); on_start возобновляет незавершённые задачи.
  Ошибка — scan_tasks.status='error' (+текст в results_data['error'],
  отдельной колонки в фиксированной схеме нет) — носитель свёртки
  error (Р-11); сбрасывается следующим сканом.
- **Параллельный скан сериала отклоняется** («уже запущен», 409 в
  старом контракте); защита и от гонки ручного с автосканом.
- **Переобработка имён перед сканом** — query renaming.reprocess
  (закрывает находку 16: сон до 600 с заменён request/reply); пока
  модуль renaming не подключён — громкий warning и пропуск.
- **fixed/rolling сохранены семантически** (fixed: замена по совпадению
  episodes — astar; rolling: замена единственного активного, при 2+
  активных не заменяем), но в явной таблице REPLACE_MODE.
- **VK-ветка**: апсерт media_items с «фантомной» защитой (1:1 со
  старым: удаляются только девственные), план — только из
  plan_status='candidate' (семантика SmartCollector), запись
  publication_date в формате хранения прод-БД; завершение —
  событие scan.plan.updated: создание download_tasks и усыновление
  файлов — зона владельца download_tasks (downloads, следующий разбор).
- **Расписание автоскана** — в самом scan: тики читают settings
  (scanner_agent_enabled, next_scan_timestamp); после полного прохода
  следующий скан назначается по опустошению конвейера — событие
  torrents.queue.changed {count} вместо опроса; состояние публикуется
  событием scan.status.changed (контракт scanner_status_update).
- **core: handle(..., concurrent=True)** — конверты топика
  обрабатываются отдельными задачами; иначе долгий скан блокировал бы
  очередь запросов модуля (урок теста на гонку; родственно находке 7).
  По умолчанию по-прежнему последовательно (порядок свёрток важен).
- Контракты будущих модулей зафиксированы фейками тестов:
  renaming.reprocess, torrents.db.active / db.deactivate_all /
  register (идемпотентен) / queue.get / queue.changed.

### Р-13. Модуль downloads (согласован 2026-06-12)
Разбор: находки 30 (полный список очереди в SSE каждые 5 с), 31 (error
ретраится только рестартом сервиса, скан плодит дубли задач), 32
(pending вне плана зависают навсегда). Решения:

- **Жизнь по событиям**: обработчик scan.plan.updated (усыновление →
  создание задач → чистка pending вне плана → пинок диспетчеру);
  диспетчер добирает задачи при создании/завершении/смене лимита —
  тик 5 с не воспроизводится. Лимит параллелизма — подписка на
  settings.changed.
- **yt-dlp асинхронным subprocess'ом** (zависание процесса не держит
  модуль), прогресс: запись в БД с троттлингом 2 с (как было) +
  событие downloads.queue.changed при изменениях (троттлинг 1 с) —
  контракт SSE download_queue_update сохранён, шторм устранён.
- **Политика ретраев**: ошибка — носитель (задача status='error' +
  элемент error, свёртка Р-11); ретрай — следующим сканом: error-задача
  заменяется новой, error-элемент снова подхватывается (старая система
  подхватывала его ТОЛЬКО рестартом). При старте reconcile возвращает
  в очередь лишь оборванные 'downloading'; error рестартом не лечится.
- **Фоновая ФС-проверка раз в 60 с не воспроизводится** (решение
  пользователя): синхронизация с диском — при каждом скане и query
  downloads.fs.sync {series_id} (пропажи → pending, находки →
  completed); gateway на этапе 5 вызывает fs.sync при открытии модалки
  статуса (там показывается актуальный список файлов).
- **Свёртка**: downloading/error по элементам плана, ready по всем
  неигнорируемым, waiting=«есть pending в плане» — семантика
  sync_vk_statuses. В агрегатор добавлено правило «waiting подавляется
  активностью (кроме ready/viewing)» — в старом коде это поведение
  было размазано по двум конфликтующим механизмам (_sync_waiting_status
  vs sync_vk_statuses), теперь детерминировано.
- Имена файлов — query rules.format_filename {series, media_item}
  (контракт зафиксирован фейком; реализация форматтера — с разбором
  renaming/rules). Обновление last_scan_time после успешной загрузки
  сохранено: command catalog.series.touch_scan_time.

### Р-14. Торрент-конвейер (согласован 2026-06-12)
Разбор стадийной машины Agent + мониторинга MonitoringAgent.

- **ИНВАРИАНТ ЯДРА (зафиксирован пользователем)**: торрент стоит на
  паузе с момента добавления и до завершения переименования файлов.
  Единственное исключение — magnet: списка файлов нет без метаданных,
  торрент запускается РОВНО на время их получения и немедленно
  ставится обратно на паузу. Любая закачка до переименования кладёт
  файлы под исходными именами и ломает зачёт существующих при
  replace-recheck (докачка только нового — сердце экономии трафика).
  В машине это закреплено: стадия awaiting_pause при любом активном
  состоянии отвечает force_pause; тесты проверяют порядок
  pause → rename → recheck → resume.
- **Стадии** — значения в БД старые (констрейнт данных + карта
  стадия→флаг Р-11), машина — чистая решающая функция
  (modules/torrents/pipeline.py). Отличие одно: переименование
  выполняется на стадии 'renaming' (раньше пустышка, работа делалась в
  конце 'awaiting_pause_before_rename') — поведение идентично.
  В семейства состояний добавлены stopped*-имена qBit 5.x.
- **Ошибка — носитель**: stage='error', задача не удаляется (текст — в
  лог: в схеме agent_tasks нет колонки), свёртка error; ретрай —
  повторной регистрацией (следующий скан). Автоскан пропускает сериал
  с активными задачами конвейера (как старый); ручной скан — нет.
- **Переименование** — query renaming.process_torrent {series_id,
  qb_hash} (зона renaming; контракт зафиксирован фейком тестов).
- **Мониторинг прогресса** — адаптивный опрос qBit (5 с при
  активности, 60 с в покое), запись в download_tasks (torrent),
  свёртка downloading/ready по семантике sync_torrent_statuses
  (+stopped*), публикация только при изменении (кэш флагов).
- **torrents.fs.verify {series_id}** — проверка файлов на диске +
  missing → recheck-задачи; вызывается сканом и при открытии модалки
  (распространение Р-13 на торренты, согласовано). Вернувшийся файл
  missing → renamed.
- Реализованы контракты Р-12: db.active (сверка с qBit),
  db.deactivate_all, register (идемпотентен: upsert по torrent_id,
  замена старого — деактивация + удаление из qBit без файлов +
  чистка torrent_files), queue.get {series_id?},
  событие queue.changed {count, tasks} (count=0 — сигнал scan).

### Р-15. Переименование и форматтер имён (согласован 2026-06-12)
Разбор RenamingAgent + filename_formatter + metadata_processor +
renaming_processor; находки 33 (single_vk создаются и удаляются без
выполнения; всё наблюдаемое поведение обеспечивала mass-переобработка)
и 34 (get_pending_renaming_task определён в Database трижды).

- **Форматтер — в rules** (чистые функции modules/rules/formatter.py,
  поведение 1:1): иерархия метаданных (овер-райды сериала > правила >
  факты БД), формат "Name_EN sNNeMM [озвучка] качество 1080p.ext",
  сезонная логика торрентов (Season NN; одно/многосезонник по
  series.season; Specials → Season 00). **Дифф-верификация: 349/349
  реальных имён фикстуры воспроизведены точно**
  (tests/test_rules_format_diff.py — имя файла и есть продукт).
  Queries: rules.format_filename {series, media_item, episode_override?,
  original_filename?} и rules.format_torrent_file {series,
  file_basename, original_path} (filename=None — сезон не определён,
  файл пропускается) — закрыты контракты-фейки Р-13/Р-14.
- **Модуль renaming**: renaming.reprocess {series_id} (Р-12; то же —
  «сохранили свойства сериала», этап 5) и renaming.process_torrent
  {series_id, qb_hash} (Р-14). Запись renaming_tasks живёт на время
  работы (носитель ошибки + is_busy-семантика); незавершённые при
  старте перевыполняются (идемпотентно); повторный запрос дожидается
  идущего (asyncio-lock по серии) вместо отказа. Событие
  renaming.finished {series_id} — контракт SSE renaming_complete.
- **single_vk не воспроизводится** (находка 33: диспетчер никогда его
  не выполнял — задачи удалялись пустыми, давая ложный is_busy);
  актуальность имён обеспечивается переобработкой при каждом скане и
  каждом сохранении свойств — как фактически и работало (трассировано:
  routes/series.py:478 создавал mass-задачу при каждом сохранении).
- **Границы владения**: rename на диске/в qBit — физика renaming;
  имена в БД пишут владельцы колонок по командам:
  downloads.item.set_filename, slicing.file.set_path (фейк до slicing),
  torrents.db.files.upsert; состав элементов — query scan.media.list.

### Р-16. Нарезка компиляций (согласована 2026-06-12)
Зона переносится семантически 1:1 (ресьюмабельность progress_chapters
оригинала хороша); отличия только инфраструктурные:

- Очередь — событийный пинок вместо опроса 5 с; по-прежнему одна
  нарезка за раз (ffmpeg грузит диск); slicing.queue.changed — при

  вечно висящий агент; ffmpeg 1800 с/глава, главы 180 с как было).

  (колонкой владеет scan); усыновление deep-adoption помечает
